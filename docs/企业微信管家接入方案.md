# 订单大看板：企业微信“管家”接入方案

## 1. 方案目标

这套方案的目标，是把“大看板”从一个纯网页管理系统，升级成一个带有企业微信入口的业务助手系统。

最终形态不是“系统去控制个人微信号”，而是：

1. 员工在企业微信里和“管家”对话。
2. “管家”把员工输入的文字或语音内容，转成项目记录、节点更新、提醒任务、待办事项。
3. 大看板后台负责存储项目、节点、提醒、人员、权限和日志。
4. 提醒结果再回到企业微信里，发给相关人员或相关群。

## 2. 总体判断

### 2.1 适合你们的企业微信方式

按你现在的业务场景，建议采用下面这套组合：

1. `企业微信智能机器人`
   作为员工和“管家”的对话入口，用来接收员工发来的文字或语音意图。

2. `大看板后台系统`
   作为业务主系统，负责项目数据、节点数据、提醒数据、权限数据和审计日志。

3. `群机器人 Webhook`
   作为群级通知通道，用于项目群、部门群、异常提醒群等场景。

4. `调度与提醒服务`
   负责定时检查哪些项目该提醒、哪些节点快超期、哪些人需要收到催办。

### 2.2 不建议采用的方式

不建议把正式方案建立在“个人微信号 + 自动操控”上。

建议正式方案统一走企业微信体系，原因是：

1. 企业内员工组织关系和权限天然就在企业微信里。
2. 后续接入机器人、群通知、组织结构、人员管理更顺。
3. 合规性、稳定性、可维护性都更高。

## 3. 企业微信能力边界

根据企业微信相关官方资料，这条路线目前可以依赖的能力主要有：

1. `企微智能机器人`支持群聊和单聊。
2. 一个应用只能发布到一个智能机器人中。
3. 用户发给机器人的消息，机器人只能在 3 分钟内完成回复，否则会被截断。
4. 企业微信对这类返回内容有字节长度限制，文档里写明每次最多支持返回 `2048` 字节。
5. 如果要用群机器人做消息推送，可以通过群里的 `Custom Message Push` 获取 `Webhook`。
6. 群机器人 `Webhook` 只有创建者可见。

这意味着产品设计上要避开几个坑：

1. 不要把机器人当成超长报告输出工具。
2. 不要把复杂审批流全部塞进单轮聊天。
3. 不要把所有提醒都压成单一机器人能力，要区分“聊天入口”和“通知通道”。

## 4. 推荐的产品架构

## 4.1 架构分层

建议拆成 5 层：

1. `企业微信交互层`
   包括企微智能机器人、群机器人，以及后续可能扩展的企微应用入口。

2. `消息接入层`
   负责接收企业微信消息、校验来源、做消息去重、解析文本和语音结果。

3. `管家编排层`
   负责把自然语言转成业务动作，比如“记事项”“挂提醒”“更新项目节点”“问我手里有哪些书”。

4. `大看板业务层`
   负责项目、节点、提醒、人员、权限、日志、配置等核心业务逻辑。

5. `数据与审计层`
   负责数据库、消息日志、提醒日志、操作日志、失败重试记录。

## 4.2 推荐的数据流

### 场景 A：员工发消息记事项

1. 员工在企业微信里给“管家”发消息。
2. 消息进入消息接入层。
3. 管家编排层识别意图，例如：
   `帮我记一下，《古籍里的日常器物》合同还没回签，下周二提醒法务陈敏。`
4. 系统解析出：
   - 项目
   - 当前节点
   - 备注
   - 提醒时间
   - 被提醒人
5. 系统返回确认语句。
6. 用户确认后，数据写入大看板。

### 场景 B：系统主动提醒

1. 调度服务扫描到某个提醒时间已到。
2. 系统读取提醒对象、项目和节点信息。
3. 如果是个人提醒，优先走企业微信个人对话或应用消息能力。
4. 如果是群提醒，走群机器人 Webhook。
5. 同时在大看板中记录提醒日志和送达状态。

### 场景 C：员工查询自己手上的项目

1. 员工发送：
   `我现在手里有哪些项目？`
2. 系统根据企业微信身份映射到内部用户。
3. 查询该用户负责项目、协作项目和待处理提醒。
4. 返回精简结果。

## 5. 功能方向规划

## 5.1 第一阶段：最小可用版

目标是先把“记得住、找得到、能提醒”跑通。

### 员工侧

1. 通过企业微信给“管家”发文字消息。
2. 记录项目事项。
3. 记录节点备注。
4. 创建提醒任务。
5. 查询自己手里的项目。
6. 查询某本书当前进度。

### 系统侧

1. 项目与人员映射。
2. 提醒人机制。
3. 节点记录与时间记录。
4. 机器人确认式写入。
5. 群级提醒推送。
6. 完整日志记录。

## 5.2 第二阶段：增强版

1. 语音输入转文字后再解析。
2. 多轮澄清，例如项目不明确时追问。
3. 提醒升级机制，比如逾期后抄送上级。
4. 节点变更自动通知相关协作者。
5. 项目日报、周报自动汇总。
6. 基于人员视角输出工作负载提醒。

## 5.3 第三阶段：正式运营版

1. 企业微信组织架构同步。
2. 角色权限细分。
3. 数据导入导出。
4. 统计报表。
5. 失败补偿与重试机制。
6. 敏感操作审计与留痕。

## 6. “管家”功能怎么按企业微信方式设计

## 6.1 聊天入口

建议由企微智能机器人承担。

适合的指令类型：

1. `记一下`
2. `提醒一下`
3. `我手里有什么`
4. `这本书到哪一步了`
5. `给谁发个提醒`
6. `把这个挂到合同节点`

不适合直接在聊天里完成的事情：

1. 大批量项目编辑
2. 复杂表格维护
3. 配置类管理动作
4. 长篇数据分析报告

这些内容仍然应该回到大看板网页后台中处理。

## 6.2 提醒通道

建议拆成两类：

### 个人提醒

目标：提醒具体某个工作人员处理下一步动作。

建议方式：

1. 第一版优先做成“管家会话内确认 + 看板内待办 + 群提醒补充”。
2. 第二版再补“面向个人的企微应用消息能力”。

说明：

企微智能机器人很适合当对话入口，但“到点主动单发提醒”的正式实现，建议后续放进企业微信应用能力里统一处理，这样更稳。

### 群提醒

目标：项目群、部门群、异常群收到提醒。

建议方式：

1. 使用群机器人 Webhook。
2. 用在项目异常提醒、即将超期提醒、每日汇总提醒。

## 6.3 身份识别

企业微信体系下，建议不要让用户手工报名字。

应该使用：

1. 企业微信身份
2. 大看板内部用户表
3. 人员与项目关系表

这样系统才能知道：

1. 谁发来的消息
2. 这个人负责哪些项目
3. 这个人有权操作哪些项目
4. 该把提醒发给谁

## 7. 你需要处理的企业微信相关事项

这一部分是你必须亲自推进或协调企业管理员推进的。

## 7.1 企业微信账号与管理权限

1. 确认你们是否已经正式使用企业微信。
2. 确认谁有企业微信管理员权限。
3. 确认是否允许新建机器人、应用和消息推送配置。

## 7.2 机器人与通道准备

1. 创建企微智能机器人。
2. 获取企业 ID。
3. 获取机器人 ID。
4. 确认机器人归属哪个业务主体或哪个团队。
5. 确认哪些群需要配置群机器人。
6. 指定谁来创建群机器人，因为 Webhook 只有创建者可见。

## 7.3 域名与回调环境

这是很关键的前置条件。

你需要准备：

1. 一个正式域名。
2. 一个可公网访问的 HTTPS 地址。
3. Nginx 或等效的反向代理服务。
4. 对应的服务器运维能力。

原因是官方资料里明确提到：

1. 企业微信机器人通过 Webhook 发送消息时，需要推送到指定 URL。
2. 不同企业需要对应不同域名。
3. 企业需要配置 Nginx，把企业微信消息抄送到腾讯云智能体。

## 7.4 企业内部人员与组织整理

1. 整理员工名单。
2. 整理部门结构。
3. 明确哪些人会用这个系统。
4. 明确哪些人只接收提醒，不直接操作系统。
5. 统一人员命名规则，避免一个人多个别名。

## 7.5 项目群与提醒群规划

1. 哪些项目要建项目群。
2. 哪些提醒发个人，哪些提醒发群。
3. 是否需要一个“异常预警群”。
4. 是否需要一个“每日汇总群”。

## 7.6 安全与责任归属

1. 谁负责管理机器人配置。
2. 谁负责保管 Webhook。
3. 谁负责管理企业微信相关密钥。
4. 谁负责处理机器人误报、漏报、重复提醒的问题。

## 8. 除了代码以外，你还需要处理的内容

这一部分通常比写代码更重要。

## 8.1 业务规则整理

你需要先把业务规则讲清楚，不然机器人很难正确理解。

至少要整理这些：

1. 标准主状态有哪些。
2. 标准节点有哪些。
3. 哪些节点允许谁修改。
4. 什么情况下需要提醒。
5. 提醒发给谁。
6. 超期怎么算。
7. 什么叫“已完成”，什么叫“暂停”。

## 8.2 人员职责划分

1. 谁是项目负责人。
2. 谁是节点负责人。
3. 谁可以当提醒人。
4. 谁是主管理员。
5. 谁负责维护基础配置。

## 8.3 使用规范

你需要给团队定使用规则，不然机器人会变得很乱。

建议明确：

1. 记事项时推荐的说法。
2. 提醒指令的标准说法。
3. 查询指令的标准说法。
4. 什么事情必须进系统，不能只停留在聊天里。

## 8.4 上线前测试准备

1. 准备一批真实但可测试的项目样本。
2. 准备一批真实提醒场景。
3. 准备一批容易歧义的输入句子。
4. 指定 3 到 5 个试运行用户先跑一轮。

## 8.5 内部推进与培训

1. 通知员工后续统一通过企业微信“管家”记事项。
2. 培训大家如何发指令。
3. 培训大家什么时候用聊天、什么时候进看板后台。
4. 指定反馈入口，收集试运行问题。

## 9. 第一版推荐实现边界

为了避免一开始做太重，建议第一版只做下面这些：

1. 企业微信智能机器人接收文字指令。
2. 识别“记事项”“挂提醒”“查项目”三类能力。
3. 大看板后台保存项目、节点、提醒、用户、日志。
4. 群机器人发送群提醒。
5. 网页后台维护用户、节点、权限、提醒规则。

暂时不要一开始就做：

1. 复杂语音理解
2. 全自动多轮推理
3. 所有人群的主动单聊推送
4. 太复杂的审批流
5. 太细的 BI 报表

## 10. 你现在最该先做的事情

如果准备正式推进，建议你先完成这 8 件事：

1. 确认企业微信管理员是谁。
2. 确认能否创建企微智能机器人。
3. 确认是否允许配置群机器人。
4. 准备一个正式域名和 HTTPS 服务器。
5. 确认谁负责 Nginx 和公网回调环境。
6. 整理员工名单、角色和负责人关系。
7. 整理项目标准流程和节点字典。
8. 明确第一版只做哪些聊天指令。

## 11. 推荐的正式落地顺序

1. 先把大看板后台的数据结构和权限模型定下来。
2. 再把企业微信用户和内部用户体系映射起来。
3. 再接入企微智能机器人，先做文字记录和查询。
4. 再接入群提醒。
5. 最后再补更复杂的提醒升级、语音和自动汇总。

## 12. 当前建议结论

最适合你的不是“微信机器人替代系统”，而是：

1. `企业微信负责入口和提醒`
2. `大看板负责业务主数据`
3. `后台负责配置、权限和审计`
4. `管家负责把自然语言变成结构化动作`

这样做，整体最稳，也最容易先跑起来。
