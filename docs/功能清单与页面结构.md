# 订单大看板：功能清单与页面结构

## 1. 产品目标

这套系统的目标，不是简单把“书名列出来”，而是解决出版业务周期长、节点多、多人协作、间隔时间久之后容易失控的问题。

第一阶段先解决 4 件事：

1. 手里当前有哪些项目，能一眼看清。
2. 每本书现在做到哪一步，能立刻定位。
3. 哪些项目卡住太久、快超期、需要跟进，能主动暴露。
4. 每个项目发生过什么，后续接手或回看时不靠记忆。

## 2. 产品定位

建议把它定义成一个“出版项目流程看板系统”，而不是单纯的状态墙。

原因是出版流程并不完全是单线推进的。比如：

1. 合同可能和校稿、印制并行推进。
2. 一校、二校、三校属于同一大阶段里的子节点。
3. 送货、尾印单往往属于后置收尾，但又需要单独跟踪。

所以系统设计上建议采用：

1. `主状态`：用于看板展示，表示项目当前所处的大阶段。
2. `关键节点`：用于项目详情，记录每个细分步骤是否完成、何时完成、谁负责。

这样既能让首页看起来清楚，也不会因为流程太细导致看板列数失控。

## 3. 流程拆分建议

### 3.1 主状态

首页看板先按大阶段展示，建议第一版使用以下列：

1. `待启动`
2. `作者沟通中`
3. `排版校稿中`
4. `样书处理中`
5. `成品制作中`
6. `合同处理中`
7. `送货处理中`
8. `尾印单处理中`
9. `已完成`
10. `已暂停`

说明：

1. `排版校稿中` 作为一个大阶段，内部再拆一校、二校、三校。
2. `合同处理中` 是否放在主流程中，第一版先保留，便于直观看到“业务收尾是否完整”。
3. 后续可以支持状态自定义，但第一版先固定，降低复杂度。

### 3.2 关键节点

每个项目详情内建议拆成以下节点：

1. 作者沟通
2. 排版完成
3. 一校完成
4. 二校完成
5. 三校完成
6. 样书完成
7. 成品完成
8. 合同完成
9. 送货完成
10. 尾印单完成

每个节点建议记录：

1. 节点状态：未开始 / 进行中 / 已完成 / 已阻塞
2. 负责人
3. 计划日期
4. 实际完成日期
5. 备注

## 4. MVP 功能清单

## 4.1 P0：第一版必须有

### 项目管理

1. 新建项目
2. 编辑项目基本信息
3. 删除或归档项目
4. 标记项目为暂停 / 完成

### 看板展示

1. 按主状态分列展示全部项目
2. 每个项目显示关键信息卡片
3. 支持拖拽或按钮切换主状态
4. 支持查看每列项目数量

### 项目详情

1. 查看项目完整信息
2. 查看关键节点进度
3. 更新节点状态
4. 记录项目备注
5. 记录最近一次跟进时间
6. 查看项目操作日志

### 查询与筛选

1. 按书名搜索
2. 按作者搜索
3. 按负责人筛选
4. 按主状态筛选
5. 按是否超期筛选
6. 按是否长时间未更新筛选

### 风险提醒

1. 标记超期项目
2. 标记长时间未更新项目
3. 标记存在阻塞节点的项目
4. 首页展示“需优先跟进”的项目列表

## 4.2 P1：第二阶段建议补充

1. 项目时间轴视图
2. 日历视图
3. 到期提醒
4. 附件管理
5. 项目评论 / 跟进记录
6. 项目标签
7. 自定义状态或字段
8. 数据导入导出

## 4.3 P2：后续增强

1. 多角色权限
2. 项目统计报表
3. 负责人工作负载统计
4. 微信或企业消息提醒
5. 合同、送货、尾印单模板化管理
6. 移动端快捷录入

## 5. 页面结构

## 5.1 页面总览

第一版建议先做 5 个核心页面 / 模块：

1. 首页看板
2. 项目详情页
3. 新建 / 编辑项目页
4. 风险提醒页
5. 基础设置页

如果想更轻，可以把“新建 / 编辑项目”做成弹窗，把“项目详情”做成右侧抽屉。这样第一版更像一个操作顺手的大屏工作台。

## 5.2 首页看板

### 目标

一眼看清“现在手里有哪些项目、各处在哪个阶段、哪些最急”。

### 主要区域

1. 顶部栏
2. 筛选区
3. 看板区
4. 风险提醒区

### 顶部栏

建议包含：

1. 系统名称
2. 新建项目按钮
3. 搜索框
4. 视图切换入口

### 筛选区

建议包含：

1. 负责人筛选
2. 状态筛选
3. 是否超期筛选
4. 是否暂停筛选

### 看板区

每一列对应一个主状态，每张卡片建议显示：

1. 书名
2. 作者
3. 当前主状态
4. 当前卡在哪个关键节点
5. 负责人
6. 最近更新时间
7. 风险标记

### 风险提醒区

建议放在首页顶部或侧边，集中展示：

1. 超过 X 天未更新的项目
2. 已超计划日期的项目
3. 当前节点阻塞的项目

## 5.3 项目详情页

### 目标

进入单个项目后，能完整看到“来龙去脉 + 当前进度 + 下一步动作”。

### 主要模块

1. 基本信息区
2. 当前状态区
3. 关键节点区
4. 跟进记录区
5. 操作日志区

### 基本信息区

建议字段：

1. 书名
2. 作者
3. 项目编号
4. 责任编辑 / 负责人
5. 合作方或客户
6. 创建时间

### 当前状态区

建议字段：

1. 当前主状态
2. 当前卡点
3. 下一步动作
4. 计划完成时间
5. 风险说明

### 关键节点区

建议使用“节点清单”或“时间轴”展示，包含：

1. 节点名称
2. 状态
3. 负责人
4. 计划日期
5. 完成日期
6. 备注

### 跟进记录区

用于记录每次沟通或推进情况，建议包含：

1. 跟进时间
2. 跟进人
3. 本次进展
4. 下次待办

### 操作日志区

自动记录：

1. 谁改了状态
2. 谁更新了节点
3. 修改发生时间

## 5.4 新建 / 编辑项目页

### 目标

让录入动作足够快，避免系统因为录入太麻烦而没人愿意用。

### 建议字段

1. 书名
2. 作者
3. 项目编号
4. 负责人
5. 当前主状态
6. 当前关键节点
7. 起始日期
8. 计划完成日期
9. 项目备注

高级字段可以折叠收起，第一版不要一次性暴露太多内容。

## 5.5 风险提醒页

### 目标

把“今天最该盯什么”单独拎出来，不让它淹没在所有项目里。

### 建议模块

1. 超期项目
2. 超过 X 天未更新项目
3. 当前节点已阻塞项目
4. 本周需要推进项目

## 5.6 基础设置页

第一版可以简化，但建议预留：

1. 状态字典
2. 节点字典
3. 负责人列表
4. 风险规则设置

## 6. 首页卡片建议字段

看板卡片信息不能太多，否则会乱。建议第一版卡片只放：

1. 书名
2. 作者
3. 当前细分节点
4. 负责人
5. 最近更新时间
6. 风险标签

可选补充：

1. 计划完成日期
2. 项目编号

## 8. 第一版设计原则

为了让系统真正用得起来，第一版建议坚持下面几个原则：

1. 先保证录入和查看顺手，不追求功能大而全。
2. 看板首页只放高频信息，不把详情硬塞进卡片。
3. 状态数量控制在 8 到 10 个以内，避免首页过宽。
4. 细分流程放进项目详情，不全部铺到首页。
5. 先把“提醒”和“最近更新时间”做好，这比花哨效果更重要。

## 9. 下一步建议

这份文档确定后，下一步可以直接进入“看板模型”阶段。建议按下面顺序做：

1. 先画首页看板原型
2. 再画项目详情页原型
3. 再补一版示例数据
4. 最后开始做 H5 页面

如果后面要落成真实产品，技术形态建议采用：

1. H5 前端页面
2. 后台服务
3. 数据库存储

第一版也可以先用前端假数据做出完整交互，再决定是否接后台。
